# P15: ghdb 앱 구축 계획

**작성일**: 2026-03-08
**목적**: 지질유산 DB 구축 사업용 별도 시스템 (화석·암석 표본 데이터베이스)

## 배경

- 2026년 지질유산 데이터베이스 구축 사업 (국가유산청 위탁)
- 화석·암석 표본의 체계적 DB 구축이 핵심
- 참여기관(국공립, 대학, 개인)의 대학원생들이 표본 정보 입력
- fsis와 유사하나 별도 시스템으로 운영 필요

## 시스템 아키텍처

### 운영 방식
- **같은 코드베이스**, 같은 Docker 이미지
- `DJANGO_SETTINGS_MODULE`로 설정 분리 → 별도 컨테이너로 운영
- 별도 DB (SQLite 파일 분리), 별도 포트

```
honestjung/fsis2026:latest (단일 이미지)
├── 컨테이너: fsis  → config.settings_fsis  → :8000 → db_fsis.sqlite3
└── 컨테이너: ghdb  → config.settings_ghdb  → :8001 → db_ghdb.sqlite3
```

## 작업 단계

### Phase 1: 모델 확장 (MuseumSpecimen 필드 추가)

현재 `body_or_trace` 필드로 체화석/생흔화석을 구분 중.
여기에 `암석` 카테고리를 추가하고, 암석 표본 전용 필드를 추가.

#### 1-1. `specimen_type` 필드 추가 (카테고리 상위 구분)

```python
# fsis/models.py - MuseumSpecimen 클래스 내
SPECIMEN_TYPE_CHOICES = [
    ("화석", "화석"),
    ("암석", "암석"),
]
specimen_type = models.CharField(
    max_length=20, blank=True, default="화석",
    choices=SPECIMEN_TYPE_CHOICES, verbose_name="표본유형"
)
```

> 기존 `body_or_trace`는 화석 내부 구분(체화석/생흔화석)으로 유지.
> `specimen_type`이 상위 구분: 화석 vs 암석.

#### 1-2. 암석 표본 전용 필드 추가

```python
# 암석 표본 정보
ROCK_CATEGORY_CHOICES = [
    ("화성암", "화성암(igneous)"),
    ("퇴적암", "퇴적암(sedimentary)"),
    ("변성암", "변성암(metamorphic)"),
]
rock_category = models.CharField(
    max_length=50, blank=True, choices=ROCK_CATEGORY_CHOICES,
    default="", verbose_name="암석 대분류"
)
rock_name = models.CharField(
    max_length=200, blank=True, verbose_name="암석명"
)
rock_name_en = models.CharField(
    max_length=200, blank=True, verbose_name="암석명(영문)"
)
rock_texture = models.CharField(
    max_length=200, blank=True, verbose_name="조직(texture)"
)
rock_structure = models.CharField(
    max_length=200, blank=True, verbose_name="구조(structure)"
)
rock_color = models.CharField(
    max_length=100, blank=True, verbose_name="색상"
)
rock_description = models.TextField(
    blank=True, verbose_name="암석 기재"
)
```

> 광물 정보는 기존 `mineral_code1~4`, `mineral_name1~4` 필드 활용.
> 지질시대, 산지, 지층명, 암상 등은 기존 공통 필드 그대로 사용.

| 크기 | specimen_size, rock_size | - | - |

### Phase 1-4. source_system 필드 (DB 통합 대비)

MuseumSpecimen과 UserActivity에 `source_system` 필드 추가.
데이터가 fsis에서 입력된 것인지 ghdb에서 입력된 것인지 기록.
나중에 두 DB를 통합할 때 출처 식별 가능.

```python
# MuseumSpecimen
source_system = models.CharField(max_length=10, blank=True, default="fsis", verbose_name='입력시스템')

# UserActivity
source_system = models.CharField(max_length=10, blank=True, default="fsis")
```

- fsis 기존 데이터: default="fsis"로 자동 태깅
- ghdb에서 표본 생성 시: `specimen.source_system = "ghdb"` 설정
- ghdb의 `get_user_obj()`에서 UserActivity 기록 시 `source_system = "ghdb"` 설정

### Phase 4: Docker 배포 구성

#### 4-1. docker-compose.yml 수정

```yaml
services:
  fsis:
    image: honestjung/fsis2026:latest
    env_file: .env
    environment:
      - DJANGO_SETTINGS_MODULE=config.settings_fsis
    ports:
      - "8000:8000"
    volumes:
      - ./data/fsis/db.sqlite3:/app/db.sqlite3
      - ./data/fsis/uploads:/app/uploads
      - ./data/fsis/staticfiles:/app/staticfiles

  ghdb:
    image: honestjung/fsis2026:latest
    env_file: .env.ghdb
    environment:
      - DJANGO_SETTINGS_MODULE=config.settings_ghdb
    ports:
      - "8001:8000"
    volumes:
      - ./data/ghdb/db.sqlite3:/app/db_ghdb.sqlite3
      - ./data/ghdb/uploads:/app/uploads
      - ./data/ghdb/staticfiles:/app/staticfiles
```

> `.env`와 `.env.ghdb`에 각각 SECRET_KEY, DEBUG, ALLOWED_HOSTS 설정.
> SECRET_KEY는 컨테이너별로 다르게 생성 권장.

#### 4-2. Nginx 프록시 설정

```nginx
# /etc/nginx/sites-available/ghdb
server {
    listen 80;
    server_name ghdb.example.com;  # 도메인 확정 후 변경

    location /static/ {
        alias /srv/fsis2026/data/ghdb/staticfiles/;
    }

    location /media/ {
        alias /srv/fsis2026/data/ghdb/uploads/;
    }

    location / {
        proxy_pass http://127.0.0.1:8001;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

현재 `gunicorn config.wsgi:application`으로 되어 있으며,

## 작업 순서

1. **Phase 1**: MuseumSpecimen에 `specimen_type` + 암석 전용 필드 추가 → `makemigrations`
2. **Phase 2**: `ghdb` 앱 생성 (startapp → forms, views, templates, urls)
3. **Phase 3**: settings 분리 (`settings_fsis.py`, `settings_ghdb.py`, `urls_ghdb.py`)
4. **Phase 4**: Docker compose 수정, 테스트
5. 기존 fsis 쪽은 새 필드가 추가되어도 기존 폼/뷰에 영향 없음 (blank=True)

## 주의사항

- `makemigrations`은 한 번만 실행, 양쪽 DB에 `migrate` 각각 적용
- fsis 기존 데이터: `specimen_type` 기본값 "화석"으로 자동 설정
- ghdb는 fsis 모델을 사용하지만, fsis의 뷰/URL은 포함하지 않음
- ghdb DB에 fsis 전용 테이블(FossilSite, UserActivity 등)도 생성됨 — 빈 상태로 무해, 모델 분리 복잡도 대비 허용
- 암석 필드 목록은 사업 진행하면서 추가 가능 (blank=True라 언제든 확장 가능)
- `.env.ghdb` 파일에 SECRET_KEY를 별도 생성하여 설정할 것

## DB 통합 대비

나중에 ghdb DB를 fsis DB로 합칠 수 있음. 같은 `MuseumSpecimen` 모델이라 스키마 동일.

### 통합 시 주의점
- **PK(id) 충돌**: 양쪽 모두 auto-increment이므로 id가 겹침. ghdb 쪽 id를 offset 시키고, `MuseumSpecimenPhoto` 등 FK도 함께 조정 필요.
- **specimen_code 규칙**: ghdb에서 입력 시 기관별 prefix를 fsis와 겹치지 않게 운영 (예: `GHDB-F-0001` 화석, `GHDB-R-0001` 암석). `specimen_code`는 수동 입력 필드(blank=True)이므로 입력 규칙으로 관리.
