# PaperMeister Sync-Centric Architecture Spec

**Date:** 2026-04-03  
**Status:** Draft  
**Scope:** Multi-source, sync-oriented corpus architecture

## 1. Purpose

PaperMeister should be treated as a system above individual bibliography or file-management tools.

It is not just:

- a Zotero importer
- a directory OCR processor
- a local PDF search app

It should instead be defined as:

**a canonical corpus layer that synchronizes with multiple external research sources and continuously builds an internal searchable and analyzable paper corpus**

This document defines that architecture.

## 2. Core Product Position

PaperMeister sits above tools such as:

- Zotero
- EndNote
- directory-oriented PDF archives
- future metadata/file sources such as RIS, BibTeX, Mendeley exports, or watched folders

Those systems remain the user's source systems.

PaperMeister becomes the system responsible for:

- ingesting from heterogeneous sources
- reconciling and merging records
- OCR and text extraction
- structured corpus generation
- analysis and assertion layers
- question-driven retrieval and summarization later on

## 3. Architectural Principle

The key architectural rule is:

**external systems are source systems; PaperMeister is the canonical corpus system**

This means:

- external systems keep their own identifiers and lifecycle
- PaperMeister stores synced snapshots and canonical internal records
- OCR and derived knowledge belong to PaperMeister
- source sync and corpus update must be first-class features

## 4. Source Types

PaperMeister should support multiple source categories.

## 4.1 Bibliographic Sources

Metadata-oriented sources, often incomplete for files.

Examples:

- EndNote
- RIS import
- BibTeX import

Typical characteristics:

- title, author, year, journal, DOI available
- attached PDF may be absent or weakly linked
- often strong for bibliography, weak for file provenance

## 4.2 File Sources

File-oriented sources, often weak in metadata.

Examples:

- local directories
- network folders
- watched archive folders

Typical characteristics:

- PDF files available
- metadata quality inconsistent
- folder structure meaningful to user

## 4.3 Hybrid Sources

Sources that provide both metadata and file attachment relationships.

Examples:

- Zotero
- future tools with both records and attachments

Typical characteristics:

- metadata plus attachment graph
- collection hierarchy
- stable external identifiers

## 5. Multi-Source Model

A single user must be allowed to register multiple sources at the same time.

Examples:

- Zotero for active library
- local folder for historical PDFs
- EndNote export for older bibliography

PaperMeister should not require the user to choose one source model globally.

Instead:

- multiple sources may coexist
- all sources contribute to one canonical corpus
- source-specific provenance remains visible
- search can operate across all sources or filtered subsets

## 6. Canonical Record Strategy

The architecture should separate source records from canonical records.

## 6.1 External Source Snapshot Layer

This layer stores what was observed from the external system.

Examples:

- Zotero item snapshot
- EndNote record snapshot
- directory file discovery snapshot

Purpose:

- preserve original identifiers and source state
- compare previous sync state with new sync state
- support change detection

## 6.2 Canonical Paper Layer

This layer represents the internal PaperMeister paper entity.

It should answer:

- what is the logical paper
- what metadata is currently trusted
- which files belong to it
- which sources claim to represent it

This is the layer searched by the user and enriched by OCR and later extraction.

## 6.3 Derived Corpus Layer

This layer contains internal data generated by PaperMeister.

Examples:

- OCR cache
- passages
- structured text
- analysis outputs
- entity candidates
- assertions

This layer is not owned by external systems.

## 7. Sync Model

Sync should be treated as a core subsystem, not a side utility.

## 7.1 Minimum Sync Responsibilities

For each source, sync must detect:

- newly added records
- metadata changes
- newly attached files
- file path changes or attachment changes
- deleted or unavailable records

## 7.2 PaperMeister Reactions

After sync detects a change, PaperMeister should decide whether to:

- create a new canonical paper
- attach the source record to an existing paper
- create or update a file record
- queue OCR
- rebuild passages or structured corpus
- refresh downstream derived outputs later

## 7.3 Sync Direction

The default roadmap should be:

1. external -> PaperMeister sync only
2. automatic internal corpus refresh after sync
3. optional selective back-sync in later versions

Early versions should avoid full bidirectional synchronization complexity.

## 8. Recommended Sync Pipeline

For each source sync run:

1. fetch or scan source state
2. store source snapshot
3. compare snapshot with previous sync state
4. classify changes as create, update, attach, detach, delete, or unchanged
5. run canonical merge/reconciliation
6. update `Paper` and `PaperFile` state
7. enqueue OCR where needed
8. update passage/structured corpus after OCR or cache load
9. mark sync result and timestamps

## 9. Reconciliation and Deduplication

The central technical problem is that the same paper may appear in more than one source.

Examples:

- same paper in Zotero and a local folder
- EndNote metadata record plus matching PDF in a directory
- same PDF present in multiple archive locations

PaperMeister should reconcile these into a single canonical paper when confidence is sufficient.

## 9.1 Strong Matching Signals

- DOI exact match
- source-native stable identifier already linked
- file hash exact match

## 9.2 Medium Matching Signals

- title exact or normalized match
- author overlap
- year match

## 9.3 Weak Matching Signals

- filename similarity
- folder similarity
- incomplete metadata similarity

## 9.4 Merge Policy

Early versions should be conservative.

- auto-merge on strong signals
- create candidate links on medium signals
- avoid auto-merge on weak signals alone

This reduces false merges that are hard to undo.

## 10. Metadata Priority Policy

Different sources have different metadata strengths.

Recommended default policy:

- trusted bibliography metadata from hybrid/bibliographic sources takes precedence over weak PDF metadata
- file-derived metadata can fill gaps but should not automatically overwrite stronger source metadata
- OCR-derived title/author guesses are fallback signals, not primary truth

Possible default preference order:

1. trusted external bibliographic source
2. linked hybrid source metadata
3. embedded PDF metadata
4. OCR-derived inferred metadata

This policy should be configurable later if needed.

## 11. Data Model Direction

The current model already contains useful foundations:

- `Source`
- `Folder`
- `Paper`
- `PaperFile`

To support sync-centric architecture cleanly, the next structural additions should likely include:

- source sync metadata
- external record snapshots or source item tables
- explicit link table between source records and canonical papers
- richer provenance fields on files and papers
- sync timestamps and statuses

Conceptually, the model should evolve toward:

- `Source`
- `SourceRecord`
- `SourceRecordFile` or equivalent attachment representation
- `Paper`
- `PaperSourceLink`
- `PaperFile`
- derived corpus tables

Exact table naming can vary, but this separation is the important part.

## 12. Source Connectors

Each source type should be implemented as a connector with a common contract.

Each connector should be able to:

- identify the source
- fetch current source state
- expose stable external identifiers where available
- expose metadata records
- expose file references or attachments where available
- report incremental change candidates if supported

Examples:

- Zotero connector
- EndNote import connector
- directory scan connector

This allows sync orchestration to remain source-agnostic.

## 13. OCR and Corpus Update Policy

When new files arrive through sync, PaperMeister should decide whether OCR is needed.

Recommended rules:

- if file hash is already known and OCR cache exists, reuse cache
- if file is new and readable, queue OCR
- if metadata changed but file did not change, skip OCR
- if file changed, treat as OCR refresh candidate

After OCR or cache reuse:

- update passages
- update structured corpus later
- keep processing idempotent

## 14. Deletion and Tombstone Policy

External deletion should not necessarily mean destructive internal deletion.

Recommended behavior:

- if a source record disappears, mark the source link inactive
- keep canonical paper if it is still supported by other sources or has derived internal value
- allow garbage collection rules later for orphaned records

This avoids accidental data loss.

## 15. User Experience Implications

The UI and CLI should reflect multi-source sync directly.

Recommended UX rules:

- user can add multiple sources
- user can sync one source or all sources
- user can search across all sources by default
- user can filter by source type or specific source
- provenance remains visible on paper detail views
- duplicate or merge candidates can be reviewed later

The product should feel like one corpus with transparent provenance, not isolated silos.

## 16. Recommended Implementation Order

The sync-centric architecture should be introduced in the following order:

1. formalize source-agnostic product language and data model direction
2. add sync metadata and source record tracking
3. refactor Zotero integration into the connector model
4. upgrade directory import into a sync-capable file source
5. add bibliographic source support such as EndNote/RIS import
6. implement reconciliation and canonical merge policy
7. trigger OCR and corpus refresh from sync events
8. later consider optional back-sync behavior

## 17. Phase Mapping

This architecture affects multiple roadmap phases.

- Phase 1: stabilize current source handling and processing behavior
- Phase 2: preserve richer structure and provenance in the canonical corpus
- Phase 3: enable source-aware corpus analysis
- Phase 4: attach assertions to canonical papers rather than source-specific copies
- Phase 5: power assistant workflows over unified multi-source corpus data

## 18. Non-Goals for the First Sync Version

- full bidirectional sync across all tools
- automatic overwrite of source-system metadata from OCR
- perfect deduplication
- graph database introduction
- always-on background daemon sync by default

The first version should focus on reliable import-sync-reconcile-update flow.

## 19. Summary

PaperMeister should be designed as:

**a sync-centric canonical corpus layer above heterogeneous research sources**

That means:

- support Zotero, EndNote-like metadata sources, and directory-based PDF archives together
- preserve source provenance
- maintain a canonical internal paper model
- treat sync as a first-class feature
- trigger OCR and corpus updates from sync events

This architecture is the right foundation for later structured corpus, assertion extraction, and research assistant capabilities.
